# 维护手册

本清单用于执行一次可审阅的内容更新。初始来源和分类不是白名单；每轮维护都应主动发现并纳入符合标准的新文章、项目、教程、产品、社区或内容形态。

## 1. 准备

- [ ] 阅读 `AGENTS.md`、`README.md`、`docs/content-guidelines.md` 和 `docs/sources.md`。
- [ ] 运行 `git status --short`，识别并保留不属于本次任务的现有改动。
- [ ] 查看 `data/resources.yaml` 的最近 `added_at`、现有 URL、标题、分类、标签和来源。
- [ ] 明确本次维护日期、搜索截止时间、任务要求的远程与目标分支；未明确推送权限时不擅自推送。

## 2. 发现候选

- [ ] 按官方、代码托管、中文社区、国际社区、研究和搜索发现分域检索；互不依赖的域可并行进行。
- [ ] 搜索最近 30 天的发布、更新、教程、项目和讨论，并记录候选原始链接。
- [ ] 回看最近 90 天，查找首次检索遗漏、后续更正、迁移公告和形成共识的重要讨论。
- [ ] 对基础教程、官方文档和经典研究做主题检索；它们不因早于 90 天而自动排除。
- [ ] 沿已收录内容的引用、参考文献、Release、相关项目和社区交叉推荐追踪至少一轮。
- [ ] 使用中英文同义词与新关键词，不把 `docs/sources.md` 或既有八类当成白名单。
- [ ] 发现优质新文章、项目、教程、产品、社区或其他内容时，加入候选而非因目录未覆盖而跳过。

## 3. 核验与筛选

- [ ] 打开原始页面，核对 HTTPS URL、标题、作者或组织、原文日期和内容正文。
- [ ] 优先官方文档、上游仓库、论文原文、作者或社区原帖；搜索页和转载仅用于发现。
- [ ] 检查项目的文档、许可证、Release 或近期维护状态；不要把 Fork 或镜像当成上游。
- [ ] 将可核验事实、作者或社区观点、厂商宣称分别标注，研究结论保留适用范围。
- [ ] 删除纯广告、低信息密度、无法公开访问、依赖登录或无法追溯原文的候选。
- [ ] 对每个 URL 做实际访问检查；记录登录墙、地区限制、动态加载或其他未能核验的原因。
- [ ] 若使用 Star、点赞、回复、下载量、价格或版本等动态数字，记录核验日期。

## 4. 去重与中文整理

- [ ] 去除跟踪参数、片段标识和末尾斜杠后，与现有 URL 比较。
- [ ] 搜索标题、项目名、论文名、事件名和中文译名，识别不同 URL 的语义重复。
- [ ] 同一事件优先保留原始或最权威来源；互补内容分别收录时写明独立价值。
- [ ] 为每条资源写自然、准确的中文标题。
- [ ] 独立撰写 40—120 个中文字符的摘要，以句号结尾，不大段翻译原文。
- [ ] 删除空泛营销词，避免把观点、厂商宣称或特定实验结果写成普遍事实。
- [ ] 日期无法从原始页面确认时，将 `published_at` 留空，不作推测。

## 5. 更新数据、来源与分类

- [ ] 在 `data/resources.yaml` 添加或更新记录，复用稳定 ID、标签拼写和适当分类。
- [ ] 检查 `id`、规范化 URL、`added_at`、`language`、`tags` 和 `featured` 是否准确。
- [ ] 新来源若已多次验证并稳定产出高质量内容，在本次改动中扩充 `docs/sources.md` 的种子、搜索式和限制。
- [ ] 新内容不适合现有分类时新增分类，不为保持旧结构而遗漏；同步扩展 `CATEGORY_LABELS`、相关测试、README 说明和必要文档。

## 6. 生成与验证

- [ ] 运行数据验证：`python -m scripts.validate`。
- [ ] 生成 README：`python -m scripts.build_readme`。
- [ ] 检查生成结果已同步：`python -m scripts.build_readme --check`。
- [ ] 运行测试：`python -m pytest`。
- [ ] 运行 Python 静态检查：`python -m ruff check .`。
- [ ] 运行空白检查：`git diff --check`。
- [ ] 再次人工打开新增链接；若使用仓库的链接检查工作流，记录失败链接及判断，避免把临时限流误当永久失效。

## 7. 审阅 diff

- [ ] 运行 `git status --short` 识别未跟踪文件，再运行 `git diff --stat`、`git diff -- .` 和 `git diff --check` 审阅整个仓库。
- [ ] 检查范围至少覆盖根目录文档、`data/`、`scripts/`、`tests/`、`docs/`、`.github/` 及状态中出现的其他路径，不得只看 README 和资源数据。
- [ ] 确认 README 人工区未被生成器覆盖，标记区与 YAML 一致。
- [ ] 确认新资源位于正确分类，最近更新顺序和精选标记合理。
- [ ] 确认生成器、验证器、测试和 GitHub 工作流没有遗漏、意外改动或与数据及文档不一致。
- [ ] 确认没有误删、重复、跟踪参数、推测日期、机械翻译、营销断言或凭据。
- [ ] 确认改动中没有混入其他任务或其他贡献者的文件。

## 8. 提交与推送

- [ ] 只暂存本次维护文件，并用 `git diff --cached` 审阅实际提交内容。
- [ ] 使用简洁命令式提交信息，例如 `content: add recent coding agent resources`。
- [ ] 提交后运行 `git status --short`，记录提交哈希和所有验证结果。
- [ ] 仅在任务明确要求、远程和分支已确认、验证全部通过时执行普通 `git push`；禁止强制推送。
- [ ] 推送后核对远程分支；若远程前进或推送失败，停止并报告，不自行覆盖历史。

## 维护报告应包含

列出新增、更新和移除的资源；新增来源或分类；检索窗口与访问限制；执行过的命令及结果；提交哈希；未解决的失效链接、日期或事实核验问题。不要把“脚本通过”表述成“所有网络链接都可用”。
