# Awesome Vibe Coding 中文资源库设计规格

## 目标

将 `scourgen/awesome-vibecoding-cn` 建设为持续更新的中文 Vibe Coding 资源入口。维护者每次发起整理任务后，由 Codex 搜索最新公开内容或处理维护者提供的链接，完成人工核验、中文整理、仓库更新、验证、提交和推送。访客打开 GitHub 仓库后，应直接通过 `README.md` 看到最新、足够丰富且可追溯的中文内容。

## 首期范围

第一版必须形成可用而非占位的资源库，覆盖以下内容：

- 最新动态与近期热点
- Agent Skills、规则文件、提示词与工作流
- 热门 GitHub 仓库与开源框架
- AI IDE、CLI Agent、应用生成器和辅助工具
- LLM 公司与 AI Coding 厂商导航
- 中文及英文精选文章（以中文摘要呈现）
- 系统学习教程、课程和实践指南
- 中文社区与国际社区热门帖子
- 研究论文、行业观点、安全与质量实践

第一版不建设静态网站，不运行无人值守爬虫，也不让模型生成的内容自动发布。后续是否增加自动化采集或独立站点，由维护者另行决定。

## 内容架构

### README 首页

`README.md` 是主要产品界面，而不是简短跳转页。顶部提供项目定位、最后更新时间、内容目录、收录原则与参与方式。正文优先展示“最近更新”，随后按用户任务组织长期资源：入门学习、工具选择、Agent 能力扩展、项目与案例、社区讨论、研究与风险。

每条公开条目至少显示中文标题、精炼中文说明、来源及原文链接。具有时效性的内容显示原文日期；工具和仓库可显示开源、免费或商业属性，但只记录能够核验的信息。

### 结构化数据

`data/resources.yaml` 是条目的单一事实来源。每条资源使用稳定 ID，并包含：

- `title_zh`：中文标题
- `title_original`：原始标题；中文原文可与中文标题相同
- `url`：规范化原始链接
- `category`：受控分类
- `source`：平台、组织或作者
- `published_at`：原文发布日期，未知时明确留空
- `added_at`：仓库收录日期
- `language`：原文语言
- `summary_zh`：独立撰写的中文摘要
- `tags`：用于筛选和交叉归类的标签
- `featured`：是否进入首页精选

生成脚本只负责确定性渲染、排序和格式化，不调用外部模型，不覆盖 README 中人工维护的项目介绍。自动生成区域使用稳定标记包围，避免误改其他段落。

### 辅助文档

- `CONTRIBUTING.md`：投稿、审核、去重和更新流程
- `docs/content-guidelines.md`：中文写作、翻译、引用、广告与质量规范
- `docs/sources.md`：信息源分层、搜索策略和访问限制
- `docs/maintenance.md`：维护者从检索到推送的完整运行手册
- Issue 模板：资源推荐与错误/失效链接报告
- `LICENSE`：MIT License

## 来源策略

### 国际来源

优先使用 GitHub 项目主页、官方产品文档、官方工程博客、论文原文、作者原帖，以及可核验的 Hacker News、Reddit 和 X 公开帖子。搜索结果页只能用于发现，不作为最终引用目标。

### 中文来源

首期覆盖 Datawhale、LinuxDo、牛客、V2EX、国内厂商技术博客和公开技术公众号文章。知乎、掘金、少数派等可作为补充。优先原帖；无法公开访问、需要登录、日期不明或疑似软文的内容降级或不收录。

### 厂商来源

覆盖主要国际与国内 LLM 公司、AI IDE/CLI、应用生成器和云平台。厂商条目用于导航和理解生态，不把营销声明改写成客观结论。

### X 与公众号限制

没有官方 API 或稳定公开页面时，通过搜索引擎发现 X 和公众号内容，再链接到可核验原文。维护流程不依赖账号 Cookie，不绕过登录墙或反爬机制，不在仓库中保存凭据。

## 维护工作流

1. 维护者发出“整理最新内容”指令，或提供一组候选链接。
2. Codex 检查上次收录时间、现有条目和来源清单。
3. 并行搜索不同来源域，优先最近 30 天，并回看 90 天内的重要遗漏。
4. 对候选项核验原始链接、作者/组织、日期、项目活跃度及内容相关性。
5. 去除重复、转载、纯广告、无法核验和低信息密度内容。
6. 撰写简洁中文标题与摘要；翻译传达原意，不伪造引语，不复制大段原文。
7. 更新结构化数据并重新生成 README。
8. 运行 schema、重复 URL、日期、排序、README 同步和链接检查。
9. 检查 diff，提交并推送到当前远程分支。

## 质量与安全规则

- 每条资源必须有可访问的原始 URL；聚合页只有在找不到原文时才可使用。
- 相同规范化 URL 只能出现一次；同一事件的多篇报道只保留信息最完整或最权威的来源。
- 中文摘要必须说明资源解决什么问题或为何值得阅读，避免“非常强大”“重磅”等空泛措辞。
- 日期不确定时不得推测；热度数据只能记录核验时可见的数字。
- 对厂商宣称、社区观点和研究结论明确区分来源，不将观点写成事实。
- 不抓取或保存账号凭据、Cookie、付费墙内容及非公开个人信息。
- AI 生成代码、安全与隐私相关内容应保留风险提示，并优先引用官方安全指南或可复核研究。

## 技术方案

采用 Python 3.11+ 和少量明确依赖：

- PyYAML 读取数据文件
- 标准库完成数据验证、URL 规范化、排序和 README 渲染
- pytest 验证 schema、唯一性、渲染稳定性和 README 同步
- Ruff 检查 Python 代码
- Lychee GitHub Action 定期或在 PR 上检查失效链接

本地提供统一命令生成和检查内容。GitHub Actions 只做质量检查，不自动采集或发布内容。

## 验收标准

- GitHub 首页 README 全中文呈现，并可直接浏览第一版丰富内容。
- 第一版至少涵盖 Skills/提示词、文章、热门仓库、LLM/厂商、教程和热门帖子六大类，并同时包含中文与国际来源。
- 所有资源都能追溯到原文，具有中文摘要，不含占位条目。
- 数据文件通过 schema 与重复检查，重复运行生成脚本不会产生 diff。
- README 生成区域与结构化数据一致。
- 贡献指南、内容规范、来源策略、维护手册、Issue 模板、许可证和 CI 齐全。
- 验证命令全部通过后，改动被提交并推送到 `origin/main`。

## 后续演进

后续可增加按月归档、RSS/Atom 来源适配、热度快照、静态站点、自动创建候选 PR 等能力。所有自动化必须先生成可审阅变更，不能绕过人工内容审核。
