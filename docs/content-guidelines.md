# 内容规范

## 收录价值

资源应至少满足一项：提供可复用的方法或教程；展示活跃、可用的项目或工具；给出有证据的案例、研究或风险分析；形成具有代表性且信息密度高的社区讨论。纯广告、关键词堆砌、无新增信息的转载和无法核验的内容不收录。

初始分类和来源不是白名单。遇到优质的新文章、项目、教程、社区或内容形态，应按同一标准评估并主动补充；现有分类无法准确表达时可以新增分类，同时保持数据、生成器、测试和文档一致。

## 中文标题

- 准确说明资源对象和主题，避免标题党。
- 以自然中文表达原意，无需逐词对应原题；产品名、项目名、论文名和必要术语可保留原文。
- 不擅自增加“官方”“开源”“免费”“最佳”等无法由原文核验的属性。

## 中文摘要

- 长度为 40—120 个中文字符，通常一至两句，以句号结尾。
- 独立说明“它是什么、解决什么问题、提供何种证据或为何值得阅读”，不写只有态度没有信息的推荐语。
- 禁用“重磅发布”“震撼来袭”“颠覆行业”“史上最强”“必看神器”“遥遥领先”等空泛营销词。
- 不得大段翻译、拼接或改写原文；摘要应是基于核验后的原创概括，不复刻文章结构，也不伪造直接引语。
- 术语首次出现时优先使用读者能理解的中文表达，确有必要再附英文；不要为了“中文化”硬译项目名。

## 事实、观点与宣称

- 可由官方文档、仓库、论文或原帖直接核验的内容才写成事实。
- 个人文章或社区讨论的判断使用“作者认为”“帖子讨论”“参与者提出”等归属语。
- 厂商关于性能、效率、安全、用户量和市场地位的说法使用“厂商称”“官方介绍”，除非有独立证据支持。
- 研究结论注明研究对象、时间、样本或适用范围；相关性、单一基准或特定早期工具的结果不能泛化为普遍因果结论。
- 无法确认的说法删除或明确标为未核实，不用含糊措辞把猜测伪装成事实。

## 日期、互动与时效性

- `published_at` 只记录原始页面可核验的首次发布日期；只有更新日期时不要假装是首发日期。
- 未知日期必须留空：在 YAML 中写 `published_at:`。不得依据搜索结果、URL 年份、Git 提交时间或当前日期推测。
- Star、点赞、回复、下载量、价格、免费额度、模型版本等会变化的信息，仅在确有判断价值时记录，并在摘要或说明中注明核验日期，例如“截至 2026-07-13”。
- 旧文章仍可因基础性或历史价值收录，但摘要要避免把过时产品状态写成现状，必要时提示迁移或后续变化。

## 来源、翻译与引用

- 优先链接官方文档、代码仓库、论文原文、项目页、作者原帖或社区原帖。搜索结果只用于发现。
- 同一内容有中英文官方版本时，选择信息更完整、链接更稳定的一版；摘要始终使用中文。
- 转载只有在原文不可得且其新增编辑价值明确时才考虑，摘要需说明来源限制。
- 不复制付费墙内容，不绕过登录或访问限制，不保存凭据、Cookie 或非公开个人信息。
- 引用原句只在其措辞本身不可替代时使用，保持短小、准确并明确归属；一般条目优先概括而非引用。

## 去重与更新

- 规范化 URL 后不得重复；移除跟踪参数、片段标识和无意义末尾斜杠。
- 同一事件的多篇文章原则上保留原始来源或信息最完整的一篇。观点、实证或后续更正互补时可以分别收录，但摘要要交代差异。
- 同一项目的主页、教程和重要公告可分别收录到不同分类，前提是各自提供独立价值，而不是重复同一介绍。
- 既有条目内容变化时优先更新原记录并保留 `id`；链接永久迁移时替换为新的规范 URL，并检查是否与其他记录重复。

## 精选标准

`featured: true` 不是热度排名。精选资源应具有长期参考价值、来源可靠、信息完整，并能代表一个重要方向。厂商广告、短期争议或仅靠互动数字走红的内容不应仅因“热门”进入精选。
