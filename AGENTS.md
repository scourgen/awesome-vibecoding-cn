# 仓库维护指南

## 项目使命与工作边界

本仓库面向中文读者持续整理可追溯、可核验的 Vibe Coding 资源。`README.md` 是读者首先看到的产品界面，`data/resources.yaml` 是资源条目的单一事实来源。每次维护都要让 README 直接可读，而不是只堆链接或把说明藏在辅助页面。

首期八个分类和 `docs/sources.md` 中的来源只是起点，不是白名单。文章以及项目、教程、产品、社区、研究等所有内容类型都要持续扩充。发现优质新文章、项目、教程、社区或尚未覆盖但确有价值的内容形态时，应主动纳入并新增必要分类；发现持续产出高质量内容的新站点、作者或组织时，应同步补充 `docs/sources.md`。发现新类别或稳定优质来源时，必须在同一改动中扩展必要的数据枚举、中文显示映射和 `docs/sources.md`，不要为了维持旧目录或旧来源表而漏掉优质内容。

## Agent 分工与 OpenCode 委派

- 主 Codex/协调 Agent 负责需求澄清、方案设计、实施计划、任务拆分、验收标准、独立审查、最终验证和发布，并对最终结果负责。
- 批量候选整理、结构化数据录入和边界明确的机械性实现，优先通过本机 OpenCode 调用 `alibaba-coding-plan-cn/qwen3.7-plus`，以减少主 Agent 的上下文消耗并加快执行。
- 委派前必须写清任务边界、允许修改的文件、禁止事项、交付格式和验证命令；不得把架构、安全、来源真伪或内容取舍等关键判断无约束地下放。
- OpenCode 的结果不能直接收录、提交或发布。主 Agent 必须复核来源、审阅完整 diff、运行本仓库全部检查，并处理审查意见。
- 若 OpenCode 或指定模型不可用，或任务需要关键架构、安全和模糊判断，主 Agent 应明确说明并直接接管，不得静默替换模型或声称已经委派。

## README-first 原则

- 先判断改动如何帮助 README 的中文读者，再决定是否收录。
- 资源条目只编辑 `data/resources.yaml`，不要手工修改 `<!-- AWESOME-VIBE:START -->` 与 `<!-- AWESOME-VIBE:END -->` 之间的生成区。
- README 标记区之外是人工区；新增项目说明、贡献入口或维护入口时放在人工区。
- 每次修改数据后运行生成命令，并审阅生成结果，而不是只确认脚本退出码。

## 一次完整更新的执行顺序

### 1. 建立基线

1. 阅读 `README.md`、`data/resources.yaml`、`docs/content-guidelines.md`、`docs/sources.md` 和 `docs/maintenance.md`。
2. 查看 `git status --short`，保留与本任务无关的用户改动。
3. 记录现有 ID、规范化 URL、最近收录日期、分类和来源，避免重复劳动。

### 2. 开放式、分域并行检索

可并行分配互不依赖的检索域，最后统一去重和定稿：

- 官方域：产品文档、更新日志、工程博客、安全公告、作者原帖；
- 代码托管域：GitHub Releases、Topics、Trending、仓库 Discussions 与维护状态；
- 中文社区域：LINUX DO、V2EX、Datawhale、牛客、掘金、知乎及厂商技术社区；
- 国际社区域：Hacker News、Reddit、公开可核验的 X 帖子及作者博客；
- 研究域：arXiv、论文项目页、研究机构博客、基准与数据集；
- 搜索发现域：主题关键词、引用追踪、相关项目、社区交叉推荐和新来源探索。

每个域优先搜索最近 30 天的新内容，再回看最近 90 天的遗漏、后续更正和重要讨论。长期有效的基础教程、官方文档或经典论文不因超出 90 天而自动排除。搜索结果页只用于发现，最终应打开并引用原始页面。

来源表不是准入清单。检索中发现优质新文章、项目、教程、社区或其他内容时要主动评估并纳入；新来源若经过多次核验且稳定产出高质量内容，应在同一改动中加入 `docs/sources.md`。

### 3. 核验候选项

优先级依次为：官方文档或公告、代码仓库或 Release、论文原文或项目页、作者原帖、可靠社区原帖。转载、聚合页和搜索摘要只能帮助定位原文，禁止作为最终收录 URL；找不到可公开访问、可核验的原始 URL 就不收录。

逐项核验：

- URL 可公开访问且使用 HTTPS，不依赖账号 Cookie 或绕过付费墙、登录墙、反爬机制；
- 标题、作者或组织、发布日期来自页面本身或可信元数据；日期未知就留空，不从 URL、搜索摘要或当前日期猜测；
- 项目仍有可用文档、代码或发布记录；历史项目只有在其历史或迁移价值明确时收录；
- 内容与 Vibe Coding、AI 编程智能体、工程工作流、质量、安全或相关生态直接相关；
- 内容具有可复述的事实、方法、案例、数据或有代表性的讨论，而非纯广告、搬运或低信息密度列表。

写作时标明证据属性：可由原始页面核验的内容写成事实；作者、社区成员的判断写成“作者认为”“讨论提出”；厂商的性能、效率、安全或市场地位说法写成“厂商称/官方介绍”，不要改写为独立事实。研究结论要保留样本、时间或适用范围，避免泛化。

### 4. 中文标题与摘要

- `title_zh` 清楚表达内容对象，不追求逐字翻译；产品名、项目名和必要术语保留原文。
- `summary_zh` 独立撰写；按 `summary_zh.strip()` 的 Python Unicode code point 总数计为 40—120（含中英文、空格与标点），说明资源解决什么问题、提供什么证据或为何值得读，以句号结尾。
- 禁用“重磅”“震撼”“颠覆”“史上最强”等空泛营销词，不大段翻译或拼接原文，不伪造引语。
- 事实、观点和厂商宣称在一句摘要内也要保持边界；互动数、Star 数等会变化的数字必须附核验日期。
- 更完整的写作和引用规则见 `docs/content-guidelines.md`。

### 5. 去重

1. 去掉 URL 中的 `utm_*`、`urlSource`、`fbclid`、`gclid`、片段标识和末尾斜杠后，与已有 URL 比较；验证器也会执行这类规范化检查。
2. 搜索相同标题、项目名、论文名、事件名及其中文译名，识别不同 URL 指向同一内容的情况。
3. 同一事件的多篇报道优先保留原始公告或信息最完整、最权威的一篇；只有观点、证据或后续进展明显互补时才分别收录，并在摘要中说明各自价值。
4. 更新既有条目时保留稳定 `id`；不要通过创建新 ID 来规避重复。

### 6. 写入 YAML

`data/resources.yaml` 顶层为 `resources` 列表。每条记录字段如下：

- `id`：稳定、唯一、可读的短横线标识；条目更新时不随标题改变。
- `title_zh`：含中文字符的中文标题。
- `title_original`：原始标题；中文原文可与 `title_zh` 相同。
- `url`：HTTPS 原始链接，移除跟踪参数并使用规范页面。
- `category`：分类键。已有 `skills`、`repos`、`tools`、`companies`、`articles`、`tutorials`、`community`、`research`；它们不是封闭枚举。
- `source`：页面显示的组织、平台或作者。
- `published_at`：原文发布日期，格式 `YYYY-MM-DD`；未知时写空值。
- `added_at`：本仓库实际收录日期，格式 `YYYY-MM-DD`。
- `language`：原文语言，如 `zh`、`en`。
- `summary_zh`：独立撰写的中文摘要，末尾使用中文或英文句号。
- `tags`：用于交叉检索的简短标签列表，复用现有拼写。
- `featured`：布尔值。仅将代表性强、信息密度高、对多数读者有长期价值的条目标为 `true`。

如新增分类，应选择稳定的英文短横线键，并同步更新 `scripts/build_readme.py` 中的 `CATEGORY_LABELS`，使 README 显示自然的中文分类名；同时更新相关测试、README 说明和必要的内容规范。若分类标签映射或其他数据枚举实际迁移到别处，以当前代码为准，但必须在同一提交中保持数据、生成器、测试和文档一致。

### 7. 生成与验证

在仓库根目录依次运行：

```powershell
python -m scripts.validate
python -m scripts.build_readme
python -m scripts.build_readme --check
python -m pytest
python -m ruff check .
git diff --check
```

`scripts.validate` 检查必填字段、类型、ID、规范化 URL、HTTPS、日期、中文标题以及摘要长度和句号；`build_readme --check` 检查 README 与 YAML 同步。网络链接可用性还需要按 `docs/maintenance.md` 人工或使用仓库配置的链接检查器核验，不能由 schema 测试替代。

### 8. 审阅 diff

先运行 `git status --short` 识别未跟踪文件，再运行 `git diff --stat`、`git diff -- .` 和 `git diff --check` 审阅整个仓库，而不是只看资源数据与 README。检查范围至少包括根目录文档、`data/`、`scripts/`、`tests/`、`docs/`、`.github/` 以及本次状态中出现的其他路径，并逐项确认：

- README 只有预期的人工区改动和由数据生成的条目变化；
- 新条目出现在正确分类，“最近更新”按已知发布日期排序，精选标记符合预期；
- 没有误删、重复事件、跟踪参数、推测日期、机械翻译或把营销宣称写成事实；
- 新分类或稳定优质来源已同步扩展生成器/数据枚举、测试、README 说明和 `docs/sources.md` 中必要部分；
- `scripts/`、`tests/`、`.github/` 等配套代码和自动化没有遗漏、意外改动或与文档不一致；
- 没有混入凭据、Cookie、付费内容、临时文件或其他任务的改动。

### 9. 提交与推送

只暂存本次维护涉及的文件，先用 `git diff --cached` 再审一次。提交信息使用简洁的命令式英文，例如 `content: add July agent workflow resources` 或 `docs: refine source discovery workflow`。提交后确认 `git status --short`，避免声称提交了仍未暂存的改动。

仅在任务明确要求推送、目标远程与分支已经确认且所有验证通过时执行 `git push`；不要强制推送，不要覆盖他人提交。若远程在本地工作期间前进，先停止并报告，按维护者决定进行同步。最终报告提交哈希、实际执行的验证及未能核验的链接或访问限制。
